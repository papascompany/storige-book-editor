Bookmoa Storige 외부사이트 플랫폼 연동 가이드

편집기는 필요한 시점에만 띄우고, PDF 검증과 합성은 플랫폼 워커로 처리합니다.

이 문서는 `Bookmoa Storige editor/storige`를 공급자 플랫폼으로 보고, bookmoa web(PHP)을 첫 운영 테넌트로 연동한 뒤 다른 인쇄 서비스까지 확장하기 위한 실행 순서입니다.

1. 현재 결론

bookmoa는 특수 하드코딩 대상이 아니라 Storige 플랫폼에 등록되는 첫 번째 운영 테넌트입니다. 다른 외부 서비스도 같은 Site/Tenant 모델로 온보딩할 수 있습니다.

이미 있음

Site/Tenant 모델

외부 쇼핑몰/앱/서비스 1개를 `sites` 행으로 관리합니다. `북모아 메인`, `점보포토`, `스튜디오북` 같은 단위입니다.

이미 있음

사이트별 인증코드

`editorAuthCode`, `workerAuthCode`를 발급하고 `X-API-Key`로 site context를 찾습니다.

이미 있음

작업/세션 site_id

편집 세션과 워커 작업에 `site_id`가 주입되어 Admin에서 사이트별 필터링이 가능합니다.

보강 필요

도메인 보안 정책

CORS, iframe CSP, postMessage, webhook allowlist를 Site 설정 기반으로 통합해야 합니다.

보강 필요

파트너 관리자

제휴사가 자기 주문/PDF/제작 상태만 처리하는 계정과 화면은 아직 없습니다.

P0 리스크

문서와 구현 불일치

URL 파라미터 snake_case/camelCase, webhook 서명 방식은 구현 전에 표준화해야 합니다.

2. bookmoa 연동가이드

기존 `docs/php연동가이드/PHP_INTEGRATION_FINAL_v3.md`와 `SYSTEM_INTEGRATION_OVERVIEW.html`을 다시 확인한 결과, 이번 플랜은 기존 PHP 연동 방식을 갈아엎는 작업이 아니라 이미 안내된 연동 계약을 Site/Tenant 모델 위에서 명확히 고정하는 작업입니다.

결론: 기존 bookmoa PHP 연동안은 유지 PHP 서버가 `X-API-Key`를 서버에만 보관하고, `shop-session` 발급 → 편집기 실행 → 편집 완료 fileId 저장 → `check-mergeable/external` → `synthesize/external` → webhook 수신 → 결과 PDF 다운로드 순서로 처리하는 큰 흐름은 바뀌지 않습니다.
유지

서버 간 인증

기존 `STORIGE_API_KEY`는 계속 사용합니다. 브라우저에 노출하지 않고 PHP 서버에서만 `X-API-Key`로 호출합니다.

유지

외부 전용 API

`/auth/shop-session`, `/files/upload/external`, `/worker-jobs/*/external`, `/files/{id}/download/external` 계약은 유지됩니다.

유지

파일 처리 방식

편집 완료 결과는 PDF URL이 아니라 `coverFileId`, `contentFileId` UUID로 저장하고, 다운로드는 서버가 프록시합니다.

확인 필요

편집기 URL 파라미터

기존 PHP 문서는 `template_set_id`, `page_count` 같은 snake_case를 안내합니다. 현재 에디터 구현은 camelCase도 사용하므로 호환 레이어가 필요합니다.

확인 필요

Webhook 서명

문서는 HMAC-SHA256을 전제로 하지만 구현은 Base64 서명 흐름이 섞여 있습니다. PHP 배포 전 한 방식으로 확정해야 합니다.

보강

Site 설정 기반 운영

북모아는 `sites`의 첫 운영 테넌트로 관리하고, 이후 외부 서비스도 동일한 인증코드/도메인/워커 옵션 규칙을 따릅니다.

bookmoa PHP 기준 표준 흐름

1. 세션 발급 PHP 서버가 `POST /auth/shop-session` 호출. 응답의 `accessToken`으로 편집기 권한을 부여합니다.
2. 편집기 실행 `template_set_id`, `order_seqno`, `page_count`, `paper_type`, `binding_type`, `return_url`을 전달합니다.
3. 완료 결과 저장 `session_id`, `cover_file_id`, `content_file_id`, `thumbnail_url`을 bookmoa 주문 DB에 저장합니다.
4. 합성 요청 결제/주문 확정 시 `check-mergeable/external` 후 `synthesize/external`을 호출합니다.
5. 결과 반영 Webhook으로 `jobId`를 받고 `GET /worker-jobs/{jobId}/output`을 서버에서 내려받아 저장합니다.

작업 전 영향 체크

영향 영역 기존 문서 기준 이번 작업 후 판단
PHP 코드 `X-API-Key` 서버 보관, `/external` API 사용, fileId 저장 큰 흐름 변경 없음. 단, HTTP 201 고정 체크보다 2xx 성공 처리 권장.
에디터 진입 URL snake_case 파라미터 예시가 PHP 문서에 있음 Storige Editor가 snake_case를 직접 수용하거나 PHP 어댑터가 camelCase로 변환해야 함.
Webhook `X-Storige-Event`, `X-Storige-Signature`, 200 응답, 최대 3회 재시도 이벤트/응답 정책은 유지. 서명 생성/검증 방식만 실제 코드와 문서 중 하나로 통일 필요.
도메인 보안 `WEBHOOK_ALLOWED_HOSTS`, CORS 도메인, API Key 비노출 수동 환경변수 중심에서 Site 설정 기반 allowlist로 확장하면 운영 영향은 줄고 온보딩이 쉬워짐.
결과 PDF `outputFileUrl`은 내부 경로라 직접 요청 금지 변경 없음. bookmoa 서버가 다운로드/저장하고 고객에게 자체 URL 또는 스트림으로 제공.
다른 외부 서비스 bookmoa 중심 문서 같은 API 계약을 `sites` 행과 별도 인증코드로 재사용. bookmoa에는 호환성 유지가 핵심.
구현 시작 전 P0 확인 항목 1) Editor URL의 snake_case/camelCase 호환, 2) webhook 서명 방식 확정, 3) bookmoa 도메인의 CORS·iframe·webhook allowlist 등록, 4) 결과 PDF와 fileId를 클라이언트에 직접 노출하지 않는지 점검해야 합니다.

3. 진행 전략 — 보완과 연동의 순서

두 저장소(Storige · bookmoa-mobile)의 보완 작업과 연동 작업을 어떤 순서로 진행할지 결정한 단락입니다. 하이브리드 진행으로 확정: Phase 0(계약 결정) → Group A 보완(병렬) → 메인 연동 → Pilot 운영 → Group B 보완(병렬).

왜 보완 먼저가 아닌가

"보완 다 끝내고 연동" 의 함정

보완 중 연동 요구사항이 보완 방향을 바꿔 두 번 작업하게 됩니다. 실데이터 없는 성능 개선은 헛수고가 되기 쉽습니다.

왜 연동 먼저가 아닌가

"연동 다 끝내고 보완" 의 함정

흔들리는 화면/엔진 위에 연동을 얹어 디버깅이 어렵습니다. 곧 바뀔 UI에 진입점을 붙이면 다시 떼야 합니다.

하이브리드 결정

인터페이스 영향만 먼저

연동 인터페이스에 영향을 주는 보완(Group A)만 먼저 처리하고, 나머지 내부 보완(Group B)은 Pilot 운영 데이터를 보고 우선순위를 잡습니다.

Group A — 연동 전 필수 보완 (두 저장소 병렬)

저장소 보완 항목 왜 연동 전인가
Storige 편집기 embed 모드 안정화 (postMessage, CSP `frame-ancestors`, `parentOrigin`) inline embed가 연동의 핵심. 불안정하면 디버깅 시 Storige 버그/연동 버그 구분이 불가능합니다.
Storige Admin 사이트 관리 기능 (`sites` CRUD, `allowedOrigins`/`frameAncestors`) Phase 1 자체가 이 작업입니다. 분리할 수 없습니다.
Storige 외부 API 응답/DTO 안정화 (URL 케이스 호환, webhook 서명) Phase 0 계약이 흔들리면 어댑터도 같이 흔들립니다.
bookmoa-mobile 상품 주문 화면 데이터 모델 정리 (`customProducts` 스키마) 어차피 `storige.*` 필드 추가가 필요. 구조가 흔들리면 cart 스키마도 같이 흔들립니다.
bookmoa-mobile 상품 상세 진입점/주문 화면 구조 정리 "편집기 열기" 버튼 위치가 흔들리면 진입점도 다시 붙여야 합니다.

Group B — Pilot 운영 후 진행 (두 저장소 병렬, 우선순위는 Pilot 데이터로 재조정)

저장소 보완 항목
Storige 워커 성능 개선 (병렬 처리, 큐 튜닝, Ghostscript 옵션)
Storige 편집기 내부 기능 추가 (도구, 효과, 정렬 옵션 등)
Storige Admin 통계/대시보드/리포트
bookmoa-mobile 주문 화면 UI/UX 개선, 결제 플로우 보강
bookmoa-mobile 부가 기능 (알림, 추천, 검색 강화 등)

전체 진행 흐름

4. 외부 도메인 제약과 해결책

외부 사이트와 Storige Editor/API 도메인이 다르기 때문에 브라우저 보안 정책을 연동 계약에 포함해야 합니다.

제약 발생 지점 공급자 대응
CORS 외부 브라우저가 Storige API를 직접 호출할 때 API Key 호출은 서버 어댑터로 제한. 필요한 브라우저 호출은 `sites.allowedOrigins` 기반 허용.
서드파티 쿠키 외부 iframe 안에서 Storige 쿠키 인증 기대 시 외부 embed는 쿠키 의존을 줄이고 `StorigeEditor.create({ token })` 또는 일회성 세션 코드 사용.
iframe / CSP 파트너 사이트 내부에서 편집기를 iframe으로 띄울 때 `frame-ancestors`를 `sites.frameAncestors` 기준으로 생성하거나 새 탭 방식을 표준으로 선택.
팝업 차단 비동기 API 호출 후 `window.open` 실행 시 사용자 클릭 직후 빈 창을 열고, 토큰 발급 후 URL을 세팅. 또는 modal iframe 사용.
postMessage 편집 완료/저장/에러 이벤트를 부모 페이지로 보낼 때 `parentOrigin`을 필수 옵션으로 받고 양쪽에서 `event.origin` 검증. `targetOrigin='*'` 제거.
Webhook allowlist PDF 검증/합성 완료 콜백 전송 시 `sites.uploadCallbackUrl` host를 검증 기준으로 사용. 수동 `WEBHOOK_ALLOWED_HOSTS` 의존을 축소.
공급자 관점의 원칙 외부 사이트 브라우저에는 API Key를 절대 노출하지 않습니다. 외부 사이트 서버가 Storige API를 호출하고, 브라우저에는 단기 편집 토큰만 전달합니다.

5. 목표 구조

Admin의 Site 설정이 외부 서비스 온보딩의 중심이고, 고객 편집 흐름과 파트너 운영 흐름은 권한을 분리합니다.

외부 사이트 bookmoa web, bookmoa-mobile, 다른 인쇄 서비스
서버 어댑터 API Key 보관, shop-session 발급, worker job 생성
Storige API siteId 주입, 세션/파일/잡 저장, webhook 발송
Editor + Worker 필요 시 편집기 실행, PDF 검증/합성 처리
파트너 처리 자기 사이트 주문 확인, PDF 다운로드, 제작 상태 관리

6. 진행 순서

아래 순서대로 진행하면 플랫폼 안전장치부터 bookmoa PHP 연동, mobile/기타 서비스 확장, 파트너 운영 포털까지 이어집니다.

1
Platform P0

Storige Admin 외부사이트 설정 확정

  • `기본설정 > 사이트`를 외부 연동의 단일 진입점으로 사용합니다.
  • `domain`, `returnUrlBase`, `uploadCallbackUrl`, `editorAuthCode`, `workerAuthCode`를 필수로 정리합니다.
  • `allowedOrigins`, `frameAncestors`, `editorLaunchMode`, 번들 URL/버전을 Site 설정으로 추가합니다.
2
Bookmoa First

bookmoa web(PHP) 1차 연동 확정

  • 기존 `STORIGE_API_KEY` 호환을 유지합니다.
  • `shop-session` → 편집기 URL → 편집 완료 저장 → `check-mergeable` → `synthesize` → webhook 저장 흐름을 확정합니다.
  • PHP 문서의 snake_case와 에디터 구현의 camelCase를 호환 레이어로 맞춥니다.
3
Security

도메인 연동 보안 계층 정리

  • CORS, iframe, postMessage, webhook host 검증을 Site 설정 기반으로 일관 처리합니다.
  • 고객 편집 권한은 `shop-session` JWT, 파트너 운영 권한은 Admin JWT + site scope로 분리합니다.
  • 새 창 방식과 iframe 방식의 표준 옵션을 문서화합니다.
4
Partner Portal

제휴사 관리자/주문처리 기능 추가

  • `PARTNER_ADMIN`, `PARTNER_OPERATOR`, `users.site_id`를 추가합니다.
  • `SiteScopeGuard`로 자기 사이트 주문/파일/잡만 접근하게 합니다.
  • 주문관리, PDF 검수/다운로드, 제작관리 화면을 역할별 메뉴로 분리합니다.
5
Adapter

외부 서비스 서버 어댑터 추가

  • 각 서비스 서버에 `/api/storige/*` 또는 PHP helper를 둡니다.
  • `STORIGE_API_BASE`, `STORIGE_API_KEY`, `STORIGE_EDITOR_URL`, `STORIGE_WEBHOOK_URL`은 서버에만 보관합니다.
  • 브라우저는 자기 서비스 어댑터만 호출합니다.
6
Catalog

상품과 템플릿셋 연결

  • `GET /api/product-template-sets/by-product?sortcode=...&stanSeqno=...` 계약을 사용합니다.
  • 상품 모델에 `sortcode`, `stanSeqno`, `templateSetId`, `allowEditor`를 저장합니다.
  • bookmoa-mobile은 `ProductEditor`와 `customProducts` 스키마를 확장합니다.
7
Editor

편집기 진입 흐름 연결

  • 주문/상품 흐름에서 필요한 시점에만 편집기를 띄웁니다.
  • iframe 방식은 `parentOrigin` 검증, 새 탭 방식은 팝업 차단 대응을 포함합니다.
  • `StorigeEditor.create(...)` embed 또는 운영 Editor URL 방식 중 하나를 표준으로 선택합니다.
8
State

편집 완료 결과 저장

  • `sessionId`, `coverFileId`, `contentFileId`, `templateSetId`, `status`를 주문/장바구니에 저장합니다.
  • 외부 서비스의 권한 판단은 자체 주문 DB 기준으로 유지합니다.
9
PDF Validation

PDF 업로드와 검증 워커 연결

  • `POST /files/upload/external`로 PDF를 업로드합니다.
  • `POST /worker-jobs/validate/external`로 검증 작업을 생성합니다.
  • `orderOptions`에는 판형, 페이지 수, 제본, bleed, 종이 두께를 정규화해 보냅니다.
10
Synthesis

주문 확정 시 합성 워커 호출

  • `check-mergeable/external`로 사전 점검합니다.
  • `synthesize/external`에 `coverFileId`, `contentFileId`, `spineWidth`, `bindingType`, `orderId`, `callbackUrl`을 전달합니다.
11
Webhook

Webhook과 결과 PDF 저장

  • `synthesis.completed`, `synthesis.failed`, `validation.*` 이벤트를 수신합니다.
  • 결과 PDF는 외부 서비스 서버가 다운로드/저장하고, 클라이언트에는 내부 storage 경로를 노출하지 않습니다.
  • webhook 서명은 구현과 문서를 맞춘 뒤 표준화합니다.
12
Pilot · Phase 7.5

1~2개 상품 실데이터 운영 (병목 식별)

  • bookmoa-mobile에서 1~2개 테스트 상품에 `allowEditor: true`를 켜고 내부 사용자 운영을 시작합니다.
  • Storige는 워커 큐 길이/평균 처리 시간/실패율, 편집기 ready→complete 소요/save 횟수/error율을 수집합니다.
  • 운영 이슈를 트래킹 시트에 기록하고 Group B 우선순위 재조정 근거로 사용합니다.
13
QA · Phase 8

사이트별 격리 E2E 검증

  • 북모아와 테스트 외부 서비스를 각각 등록하고 서로 다른 키로 작업을 만듭니다.
  • Admin 사이트 필터로 작업/세션이 분리되는지 확인합니다.
  • 파트너 계정이 다른 사이트 주문/파일/job 상세에 접근하면 403인지 검증합니다.
  • Pilot 운영(Phase 7.5)과 병행 가능. 단, 회귀 시나리오 6~12(inline embed) 통과 없이 운영 배포 금지.
14
Group B · Phase 9

Pilot 데이터 기반 내부 보완 (두 저장소 병렬)

  • Storige: 워커 성능 개선(병렬·큐 튜닝·Ghostscript 옵션), 편집기 내부 기능, Admin 통계/대시보드.
  • bookmoa-mobile: 주문 화면 UI/UX 개선, 결제 플로우 보강, 부가 기능(알림·추천·검색).
  • 모든 PR은 Phase 8 회귀 시나리오 통과 후 머지. 연동 인터페이스 변경 금지.

8. 제휴사 관리자 기능 개발 순서

파트너 계정은 편집기 고객 세션과 분리된 운영 권한입니다. 자기 사이트 주문, 파일, 워커 작업만 처리해야 합니다.

API / DB

  1. `PARTNER_ADMIN`, `PARTNER_OPERATOR` 역할 추가
  2. `users.site_id` 또는 `site_users` 추가
  3. `SiteScopeGuard` 추가
  4. files, edit_sessions, worker_jobs 상세 권한 강화
  5. `/partner/orders` API 추가

Admin UI

  1. 역할별 메뉴 분리
  2. 파트너 주문관리 화면
  3. PDF 검수/다운로드 화면
  4. 제작상태 변경 화면
  5. 자기 사이트 고정 필터

운영 정책

  1. 파트너 계정 발급/비활성화
  2. PDF 다운로드 감사 로그
  3. 상태 변경 이력 저장
  4. 다른 사이트 접근 차단 테스트
  5. 장기적으로 사이트별 통계 대시보드

9. PDF 워커 처리 흐름

여기서 worker는 Cloudflare Worker가 아니라 NestJS + Bull + Redis 기반 PDF 처리 서비스입니다.

1. PDF 업로드 `/files/upload/external`
2. 검증 Job `/worker-jobs/validate/external`
3. 합성 사전점검 `/worker-jobs/check-mergeable/external`
4. 합성 Job `/worker-jobs/synthesize/external`
5. 결과 저장 Webhook 수신 후 PDF 다운로드/저장
사이트별 분리 모든 외부 worker 호출은 `X-API-Key`로 siteId를 찾아 `worker_jobs.site_id`에 저장해야 합니다.

  • Admin 사이트 필터 동작 확인

  • 필요 시점에만 편집기 실행

12. 최종 주의점

가장 먼저 정리할 것 URL 파라미터와 webhook 서명은 외부 연동의 계약입니다. bookmoa PHP 연동 전에 문서와 실제 구현을 반드시 맞춰야 합니다.
파트너 포털은 별도 Phase 제휴사 관리자 기능은 보안 범위가 크므로 편집기 연동과 분리해 개발하되, 지금부터 모든 API에 `site_id` 권한 모델을 일관되게 적용해야 합니다.
보완 작업은 Group A와 Group B로 분리 운영 Group A(연동 인터페이스에 영향)는 메인 연동 시작 전, Group B(내부 개선)는 Pilot 운영(Phase 7.5) 후로 미룹니다. 분류를 섞어 동시에 진행하지 않습니다.
Group B 우선순위는 Pilot 데이터로 재조정 사전 추정만으로 우선순위를 고정하지 않습니다. Pilot 메트릭(큐 길이, 처리 시간, 실패율, 사용자 피드백)을 본 뒤 결정합니다.
Group A 병렬 진행 중 매주 계약 정합성 점검 두 저장소가 동시에 진행하므로 Storige API DTO와 bookmoa-mobile 어댑터 시그니처가 Phase 0 계약과 일치하는지 매주 확인하고, 어긋나면 즉시 동기화합니다.
