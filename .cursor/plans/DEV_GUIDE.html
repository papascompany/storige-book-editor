Storige 인수 개발 가이드

북모아(Bookmoa) 쇼핑몰용 에디터 + PDF 합성 시스템을 물려받은 당신을 위한 단계별 로드맵. 이 문서 하나만 보면 — 코드 이해 → DB 복구 → 기능 보완 → Docker 배포까지 이어서 할 수 있습니다.
React · Fabric.js NestJS · TypeORM MariaDB · Redis Bull · pdf-lib · Ghostscript Docker Compose

1. 이 가이드를 쓰는 법

이 페이지는 .cursor/plans/ 에 있는 7개 마크다운 문서를 한 장에 시각화한 것입니다. 왼쪽 사이드바에서 섹션을 이동하고, 체크박스로 내가 어디까지 했는지 추적하세요. 체크 상태는 브라우저에 자동 저장됩니다(다른 기기로 넘어가려면 "내보내기" 버튼).

읽기 순서: 위에서 아래로 순서대로. 건너뛰지 마세요. 모르는 용어가 나오면 하단 부록. 용어집을 먼저 확인.

이 시스템을 처음 여는 당신이 해야 할 일 (한 문장)

"DB가 깨진 채로 시작합니다" — 실제 코드는 DB 테이블 23개를 요구하는데 초기화 스크립트는 10개만 만들어줍니다. 이 상태로 개발 서버는 돌지만, 운영 배포 시 500 에러가 쏟아집니다. P0-A부터 해결해야 합니다.

2. 5분 요약 — 이게 뭐 하는 시스템인가

한 줄 정의

Storige = 북모아 쇼핑몰에 붙는 "웹 에디터 + PDF 합성" 시스템. 고객이 브라우저에서 템플릿을 편집하면 → 서버가 인쇄용 PDF를 생성합니다.

구성 4개

apps/editor :3000
고객용 캔버스 에디터 (React + Fabric.js + Zustand)
apps/admin :3001
운영자 템플릿 관리 (React + Ant Design)
apps/api :4000
REST API 게이트웨이 (NestJS + TypeORM + MariaDB)
apps/worker :4001
PDF 처리 워커 (NestJS + Bull + pdf-lib + Ghostscript)

데이터 흐름 (핵심)

  1. 외부 쇼핑몰 PHP → 고객을 editor로 임베드 진입시킴
  2. editor가 캔버스 내용을 edit_sessions에 저장
  3. 편집 완료 → api가 worker_jobs에 작업 등록 + Bull 큐에 enqueue
  4. worker가 pdf-validation → pdf-conversion → pdf-synthesis 순서로 처리
  5. 완료 시 api가 callback/웹훅으로 결과 통지

8. 단계별 상세 체크리스트

P0-A DB 스키마 일치화 (13개 테이블 추가)
왜 제일 먼저? 이게 안 풀리면 편집 완료→검증 체인 전체가 500으로 막힙니다. 개발 서버는 synchronize: true로 덮어써서 안 보이므로 운영 직전에야 드러나는 함정입니다.

절차 (방법 b: 수동 SQL 편집)

  1. .cursor/plans/P0A_START_HERE_결정가이드.md 읽고 (b) 선택 확정
  2. .cursor/plans/P0A_DB_SCHEMA_FIX.md의 완성본 SQL을 docker/mysql/init.sql로 교체
  3. 기존 DB 볼륨 삭제: docker-compose down -v
  4. 깨끗하게 재기동: docker-compose up -d mariadb
  5. 검증: docker exec -it storige-mariadb mysql -ustorige -p storige -e "SHOW TABLES" → 23개 이상
  6. API 띄워서 어드민에서 템플릿셋 1개 생성 성공 확인
  7. 커밋: fix(db): init.sql과 엔티티 23개 일치화 (P0-A)
P0-B Dockerfile || true 제거

3개 파일, 4군데에서 빌드 실패를 삼키는 || true를 제거합니다. 빌드가 실제로 깨져도 오류 없이 성공처럼 보이는 상태.

# 위치
docker/api/Dockerfile:23
docker/worker/Dockerfile:35
docker/editor/Dockerfile:23
docker/editor/Dockerfile:24

# 수정 전
RUN pnpm --filter @storige/types build || true

# 수정 후
RUN pnpm --filter @storige/types build

검증: docker-compose build --no-cache 로 전체 재빌드. 각 앱이 문제없이 빌드되어야 함. 실패하면 진짜 버그가 있던 것 — 수정 후 재시도.

P0-C Worker callback 경로 통일

현재: validation.processor.ts:119는 /worker-jobs/external/:id/status(올바름)를 호출하지만, conversion.processor.ts:102와 synthesis.processor.ts:854는 /worker-jobs/:id/status(비-external)를 호출. 서버는 전역 JwtAuthGuard만 있어 401로 조용히 실패합니다.

해결 A (처리기 수정 · 추천): 두 processor의 URL을 /worker-jobs/external/:id/status로 교체. Validation처럼 동작.

해결 B (API 수정): 컨트롤러 :id/status에도 동등한 @Public()+ApiKeyGuard 경로를 추가. 호환성 유지.

P0-D Worker .env.example 보완

apps/worker/.env.example에 WORKER_API_KEY와 API_BASE_URL 추가:

# apps/worker/.env.example 끝에 추가
# Worker → API 통신
API_BASE_URL=http://localhost:4000/api
WORKER_API_KEY=test-api-key

주의: 실제 .env에도 같은 값을 넣되, WORKER_API_KEY는 반드시 루트 .env의 API_KEYS 콤마 목록에 포함된 값이어야 합니다.

부록. 용어집 (초보자용)

모노레포 (monorepo)
여러 앱·패키지를 한 저장소에서 관리. pnpm workspaces + Turborepo 사용.
Bull 큐
Redis 기반 작업 큐. API가 producer, worker가 consumer.
TypeORM synchronize
엔티티 정의에 맞춰 DB 스키마를 자동 수정. dev는 편하지만 운영엔 절대 금지.
엔티티 (Entity)
TypeScript 클래스로 DB 테이블을 선언한 것. apps/api/src/*/entities/*.entity.ts.
가드 (Guard)
NestJS에서 엔드포인트 접근 통제 미들웨어. JwtAuthGuard, ApiKeyGuard, RolesGuard.
@Public() 데코레이터
전역 JwtAuthGuard를 이 엔드포인트에서만 우회시키는 표시.
플러그인 (Canvas)
Fabric.js 기반 에디터의 기능 단위. packages/canvas-core/src/plugins/.
Ghostscript
PDF 후처리 (CMYK 검증 등)용 외부 도구. worker 컨테이너 안에 설치됨.
Sharp
Node.js 이미지 리사이즈/변환 라이브러리. 썸네일 생성에 사용.
HMAC
공유 비밀키로 만드는 암호학적 서명. 현재 webhook은 base64인데 HMAC-SHA256으로 승격 권장 (P7).
멱등성 (Idempotency)
같은 요청을 여러 번 받아도 1번 처리된 것과 동일. worker_jobs의 (session_id, pdf_file_id, request_id) UNIQUE 키로 보장.
