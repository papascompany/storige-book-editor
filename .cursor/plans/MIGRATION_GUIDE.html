Storige 신규 인프라 구축 가이드

이전 개발자에게 어떤 권한도 받지 못한 상황에서, 북모아 PHP 운영 서버는 손대지 않고 새 인프라를 처음부터 구축해 같은 서비스를 다시 띄우는 단계별 follow-along 매뉴얼.
전제 시나리오: GitHub 권한 ❌ · Vercel 권한 ❌ · 서버 SSH ❌ · DB 백업 ❌ · 도메인 양도 ❌  |  유지: 북모아 PHP 운영 서버 (콜백/연동 대상)

A. 이 가이드를 누가 쓰나

당신이 다음 중 하나라도 해당하면 이 가이드입니다:

  • 이전 개발자와 연락이 끊겨 GitHub·Vercel·서버 접근권을 받지 못한다.
  • 회사 내부 정책상 운영 인프라를 본인 명의로 처음부터 다시 구축해야 한다.
  • 이전 인프라가 문제가 많아 아예 새로 시작하는 게 빠르다고 판단했다.
  • 북모아(또는 외부 쇼핑몰) PHP 측 협조는 가능하다 — config 한두 줄 변경 정도.
이 가이드의 한계 이전 운영 데이터(템플릿셋·라이브러리·편집 세션·파일)는 가져올 수 없으면 잃습니다. 이 부분은 기술이 아니라 협상 영역이므로 §C-3에서 별도로 다룹니다.
이 가이드와 같이 봐야 하는 자료 기본 개발 가이드는 DEV_GUIDE.html입니다. 여기 안 적힌 것(P0-A DB 스키마 수정, P1~P7 코드 보완 등)은 거기서 진행하세요. 이 문서는 인프라 구축에만 집중합니다.

S. 작업 범위 — 정확히 무엇을 하는 가이드인가

한 줄 정의

코드는 그대로 복사 · 환경(인프라/계정)만 본인 명의로 새로.
새로 코딩하는 게 아니라, "같은 소스를 본인 명의의 새 인프라에 다시 띄우는" 작업입니다. 추가 코딩은 P0-A DB 스키마 적용 외에는 이전 자체에 없습니다.

두 트랙으로 분리해서 보기

이 가이드(A)와 DEV_GUIDE.html(B)는 서로 다른 작업을 다룹니다. 섞지 말고 트랙별로 진행하세요.

섞지 말 것 A·B를 동시에 같은 브랜치에서 하면 어디서 깨졌는지 분간이 안 됩니다. A를 먼저 끝내서 운영을 본인 인프라로 옮긴 뒤 → B를 그 위에서 브랜치/PR로 진행하세요. 단, P0-A(DB 스키마)는 A의 운영 시작 전 필수이므로 A의 Phase 2-3에 묶어 두었습니다.
A 트랙을 먼저 하는 이유
  • 이전 개발자 의존성을 끊어야 본인이 자유롭게 수정 가능
  • 코드 변경 없이 인프라 이전만으로 "내 운영"이 됨
  • B의 코드 변경은 PR·브랜치로 안전하게 가능
B를 미루면 안 좋은 점
  • P0-C 미해결 → Worker 상태가 일부 안 바뀌어 보임
  • P1 미해결 → 편집 완료 후 검증이 안 돎
  • 그래서 A 컷오버 직후 1~2주 안에 B도 진행 권장

헷갈리는 용어 정리 — "개발 환경"의 두 가지 의미

용어의미이 작업에서
로컬 개발 환경 본인 노트북/PC에서 pnpm dev 돌리는 환경 이미 본인 것이라 변경 불필요. git remote만 새 레포로 교체
운영 서버 환경 VPS + Vercel + 도메인 + DNS 이 가이드의 핵심 대상. 통째로 본인 명의로 새로 만듦
스테이징(선택) 운영 직전 검증용 별도 환경 VPS 1대 더 띄워서 같은 절차 반복하면 됨 (필수 아님)
정확하게 표현하면 "로컬 개발 환경 + 운영 서버 환경을 본인 명의로 분리"이고, "코드는 그대로 가져가서 새 인프라에 띄움"입니다. 이 가이드는 후자(운영 서버 환경)에 집중합니다.

자주 나오는 오해 4가지

  1. "새로 만든다" ≠ 코드를 다시 짠다는 뜻이 아님. 인프라/계정만 신규.
  2. "마이그레이션"은 보통 DB 마이그레이션이지만, 여기선 운영권 이전(Migration of ownership)을 가리킴. 코드 마이그레이션 아님.
  3. "포크 레포 폐기" = 즉시 삭제 아님. 새 레포 안정화 후 GitHub Settings에서 Archive가 안전.
  4. "P0-A는 코드 보완 아니냐" — 맞지만 운영 첫 가동 전 필수라 인프라 트랙(A)에 포함. 그래서 A 트랙에 들어 있는 유일한 코드 변경.

B. 전체 그림

Before — 지금 운영되고 있는 모습 (이전 개발자 인프라)

After — 우리가 새로 만들 모습

핵심 인사이트 3개

① PHP는 baseURL만 바뀜

PHP의 config.php에서 API 엔드포인트 도메인 한 줄만 새 도메인으로 교체. 코드 본체는 손대지 않음.

② 코드는 그대로 재사용

현재 포크한 소스를 새 GitHub 레포로 그대로 push. docker-compose.yml도 그대로 쓰면 됨.

③ 데이터는 협상 영역

DB·파일을 가져오면 좋고, 못 가져오면 빈 DB로 시작하고 어드민 시드부터. 기술 외 영역.

C. 사전 준비물

C-1. 사야 할 것 (총 비용 ~월 5만원)

C-2. 구해야 할 정보 (협조 요청)

PHP 운영자(또는 회사 IT 담당자)에게 한 번에 모아서 요청 — 템플릿은 §M에.

C-3. 데이터 인수 가능성 트리

운영 DB·파일을 가져올 수 있나? (펼쳐서 의사결정)
경우대처잃는 것
덤프를 받음새 서버에 import → 거의 무손실 인수없음
읽기만 가능본인이 직접 mysqldump 실행 후 import없음
둘 다 안 됨빈 DB로 시작. 어드민에서 템플릿셋 다시 등록. PHP가 보내는 신규 주문부터 처리이전 템플릿·라이브러리·진행중 세션·이미 합성된 PDF
스크린샷만 가능화면 보고 손으로 다시 만들기 (소량 템플릿이면 가능)대량 데이터·바이너리 파일
데이터를 못 가져올 경우 "이전 시스템과 새 시스템의 컷오버 시점에 진행 중이던 주문은 이전 시스템에서 처리 완료"로 PHP 측과 합의해두세요. 그래야 신규 주문부터만 새 시스템이 받습니다.

        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /storage/ {
        alias /app/storage/;
        expires 7d;
    }
}

T. 트러블슈팅

docker compose build가 worker에서 실패

node-canvas의 cairo·pango 빌드 실패가 가장 흔함. 메모리 4GB 미만 VPS에서 OOM. 4GB 이상 + swap 추가:

sudo fallocate -l 4G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab
HTTPS 연결 시 "유효하지 않은 인증서"

DNS 전파 안 됐거나 certbot이 80 포트를 못 잡음. 컨테이너의 nginx가 80을 쓰고 있으면 충돌:

docker compose stop nginx
sudo certbot certonly --standalone -d api.<도메인>
docker compose start nginx
PHP에서 호출 시 401 Unauthorized

99% API_KEYS 불일치. PHP가 보내는 헤더 확인:

docker compose logs api | grep -i "401\|api.key\|unauthorized"

또는 curl -H "X-API-Key: <키>" https://api.<도메인>/api/product-template-sets/by-product?sortcode=001001001로 직접 테스트.

Vercel 빌드가 Cannot find module @storige/types

Build Command에서 types 빌드를 안 했음. 위 §3-3 참고. cd ../..로 워크스페이스 루트에서 시작해야 함.

Worker가 큐를 안 집어감

apps/worker/.env의 REDIS_HOST가 localhost면 컨테이너에서 못 찾음. Docker Compose 환경에선 서비스명 redis 사용:

REDIS_HOST=redis
REDIS_PORT=6379
어드민 로그인 후 401이 계속 뜸

JWT_SECRET이 컨테이너 재시작마다 바뀌면 토큰이 무효화됨. .env에 고정값으로 박혀 있는지 확인. 또는 클라이언트 캐시 삭제.

편집 완료 후 callback이 안 옴

3가지 의심: ① useWorkSave.ts:666 TODO 미구현(P1) ② callback URL이 잘못 등록됨 ③ 새 서버에서 PHP로의 outbound가 막힘. docker compose logs api | grep -i webhook로 확인.
