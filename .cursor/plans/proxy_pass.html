Storige API 인프라 전환 안내

북모아(bookmoa) 사이트의 storige 에디터/워커 API가 새 인프라(api.papascompany.co.kr)로 이전되었습니다. PHP 코드는 변경 없이, 북모아 서버의 Apache 설정 2곳만 변경하면 전환이 완료됩니다.

한 줄 요약 — 변경할 것

북모아 서버(Ubuntu 22.04 + Apache + Nginx)의 Apache vhost 설정에서 다음 두 가지를 변경하시면 됩니다. 나머지는 모두 기존 그대로 (PHP 코드, Nginx, .htaccess, DB, 환경변수 등 전부 변경 없음).

① ProxyPass URL 1줄 변경 북모아 고객용 편집기 페이지(/editor/*)가 사용하는 reverse proxy 대상 ProxyPass /storige-api/ https://api.papascompany.co.kr/api/
② SetEnv STORIGE_API_URL 1줄 변경 nimda 관리자 모듈의 storige_common.php가 사용하는 API base URL SetEnv STORIGE_API_URL "https://api.papascompany.co.kr/api"
✅
STORIGE_API_KEY 변경 없음 — 기존 키 sk-storige-l3YV...를 새 인프라가 그대로 인식하도록 등록 완료. PHP 코드, .htaccess, DB는 0줄 변경.
🔁
이 한 번의 ProxyPass 변경으로 자동 전환되는 모든 페이지:
✓ /editor/index.php · /editor/editor.php · /editor/callback.php · /editor/worker-test.php · /storige-admin/ · nimda 모듈의 storige_common.php
모두 같은 /storige-api 경로를 사용하므로 ProxyPass 한 줄 변경으로 일괄 적용. 정적 admin SPA 빌드물도 다시 빌드할 필요 없음.

01

왜 이 변경이 필요한가

papascompany 측에서 기존 storige 시스템을 인수받아 새 클라우드 인프라로 마이그레이션하는 작업을 진행했습니다. 마이그레이션의 핵심 원칙은:

🎯
"기존 PHP 연동 방식은 절대 변경하지 않는다" — 북모아의 PHP 코드, DB, nginx, .htaccess는 일체 손대지 않고, storige API/Editor/Worker 시스템만 새 인프라로 옮긴다는 원칙입니다.

변경 배경

구분옛 인프라 (기존)새 인프라 (이전 후)
위치 북모아 운영 서버 내부 (Node.js v20 위에 storige 가동)
접근 IP: http://58.229.105.98:4000		 papascompany 별도 클라우드 (Vultr Seoul VPS, Docker)

운영 책임 인수 전 storige 개발자 (blueamethyst) papascompany 인수팀

자동 백업 설정 미상 매일 03:00 KST cron (DB + 파일 + .env, 7일 보존)

왜 PHP 측 변경이 최소화되는가

  • API 계약(스펙) 동일 — 새 인프라는 옛 인프라와 동일한 endpoint/path/payload schema를 그대로 제공. bookmoa PHP가 호출하는 모든 endpoint(/auth/shop-session, /files/upload/external, /edit-sessions/external, /worker-jobs/synthesize/external 등)는 같은 입력에 같은 응답.
  • API Key 그대로 — 기존 nimda PHP가 쓰는 STORIGE_API_KEY(sk-storige-l3YV...)를 새 인프라 측 API_KEYS 환경변수에 추가 등록 완료. 따라서 PHP 측 키 값도 변경 불필요.
  • Webhook payload 동일 — synthesis.completed/failed 이벤트의 field 구조(jobId, orderId, status, outputFileUrl, outputFiles, outputFormat, errorMessage, timestamp)는 그대로. X-Storige-Signature, X-Storige-Event 헤더도 동일.
  • 차이는 호스트 주소뿐 — 따라서 PHP가 storige API를 어떤 호스트로 호출할지만 바꾸면 됩니다 (Apache의 ProxyPass + SetEnv 두 군데).

검증 결과 (papascompany 측에서 사전 완료)

  • ✓
    새 인프라가 옛 인프라와 동일 응답 schema 검증 기존 nimda PHP 키 sk-storige-l3YV...로 양쪽 호출 시 동일 schema ({"success":true,"data":[...]})
  • ✓

  • ✓
    합성 결과 webhook 정상 송신 X-Storige-Signature 포함, payload schema 옛 인프라와 동일

예시 vhost (참고)

실제 vhost 설정 위치와 형태는 환경마다 다릅니다. 아래는 일반적인 패턴 예시입니다.

# /etc/apache2/sites-enabled/bookmoa.conf (예시)
<VirtualHost *:443>
    ServerName bookmoa.noriter.co.kr
    DocumentRoot /var/www/html

    # [기존, 변경 X] PHP/SSL/문서 루트 설정 등은 그대로
    SetEnv INC /var/www/html/inc
    SetEnv SELL_SITE bookmoa
    SetEnv SiteHome /var/www/html

    # --- ① ProxyPass 변경 ---
    ProxyPass /storige-api/ http://58.229.105.98:4000/api/
    ProxyPass /storige-api/ https://api.papascompany.co.kr/api/
    ProxyPassReverse /storige-api/ http://58.229.105.98:4000/api/
    ProxyPassReverse /storige-api/ https://api.papascompany.co.kr/api/

    # --- ② SetEnv 변경 ---
    SetEnv STORIGE_API_URL "http://58.229.105.98:4000/api"
    SetEnv STORIGE_API_URL "https://api.papascompany.co.kr/api"

    # [기존, 변경 X] API 키는 그대로
    SetEnv STORIGE_API_KEY "sk-storige-l3YVceH0sB739pgTfxRAxZAmLJROcMtgdKPIDYdVG0g"

    # [기존, 변경 X] 편집기 URL 있다면 새 값으로 (있을 때만)
    SetEnv STORIGE_EDITOR_URL "https://editor.papascompany.co.kr"

    # [기존, 변경 X] webhook 검증 헤더
    SetEnv STORIGE_WEBHOOK_VERIFY_HEADER "X-Storige-Signature"
</VirtualHost>
💡
HTTPS 기반 ProxyPass에 필요한 모듈 — Apache가 HTTPS 백엔드로 ProxyPass하려면 mod_ssl이 활성화돼 있어야 합니다. 대부분 환경에 이미 활성화돼 있지만, 혹시 비활성화 상태라면 a2enmod ssl proxy proxy_http 후 reload가 추가로 필요합니다.

  • EditSession 완료 응답 (status: complete)

3

nimda "에디터파일" 탭 동작

nimda 관리자 페이지에서 위에서 만든 주문(orderSeqno)의 상세 팝업 → "에디터파일" 탭 클릭.

확인 사항:

  • AJAX로 load_storige_files.php 호출 → 200 응답
  • 응답 schema가 files{cover, content, merged} 포함
  • 표지/내지 PDF 다운로드 링크 클릭 시 정상 다운로드

⚠️ 합성 잡(POST /worker-jobs/synthesize/external)이 PHP 결제/접수 흐름에서 자동 호출된다면, 그 단계도 정상인지 (callbackUrl로 webhook 수신) 확인.

Step 3. 운영(Production) 컷오버

⏰
전제: Step 2의 staging 검증 4종이 모두 통과한 후 진행. 가급적 트래픽 최저 시간대(한국 새벽 03:00~04:00 KST 권장)에 수행.

컷오버 절차

1

운영 vhost 백업 + 변경

소요: 2분 · 다운타임: 0
# 운영 vhost 파일 백업
sudo cp /etc/apache2/sites-enabled/bookmoa-prod.conf \
        /etc/apache2/sites-enabled/bookmoa-prod.conf.bak.$(date +%Y%m%d_%H%M%S)

# Step 1과 동일하게 ProxyPass + SetEnv 두 줄 새 값으로 변경
sudo nano /etc/apache2/sites-enabled/bookmoa-prod.conf
2

문법 검증 + graceful reload

sudo apachectl -t                # Syntax OK
sudo apachectl graceful          # 다운타임 0
3

첫 5분 모니터링

# 변경 즉시 확인
curl -sS https://bookmoa.com/storige-api/health   # 새 인프라 uptime이어야 함

# Apache 에러 로그
sudo tail -f /var/log/apache2/error.log | grep -i storige

# PHP 에러 로그
sudo tail -f /var/log/apache2/php-error.log
4

임시 주문 1건 테스트

운영 환경에서 본인 계정으로 테스트 주문 1건을 진행해 전체 흐름이 정상인지 확인.

5

24시간 모니터링

  • API 에러율 0% 유지
  • worker 큐 적체 없음 (papascompany 인수팀이 모니터링)
  • webhook 수신 성공률 100%
  • nimda 측 PDF 다운로드 정상
🎉
24시간 무사고면 컷오버 완료. 옛 storige(58.229.105.98:4000)는 더 이상 트래픽을 받지 않으므로 정리 가능.

롤백 절차 — 만약 문제 발생 시

↩️
옛 storige(58.229.105.98:4000)가 그대로 가동 중이므로 즉시 롤백 가능. 북모아 운영 서버에서 옛 storige를 종료하거나 삭제하지 않은 상태이므로, vhost 한 줄을 다시 옛 값으로 되돌리면 즉시 복구됩니다. 
# 1. 백업 파일로 즉시 복원
sudo cp /etc/apache2/sites-enabled/bookmoa-prod.conf.bak.YYYYMMDD_HHMMSS \
        /etc/apache2/sites-enabled/bookmoa-prod.conf

# 2. 문법 검증 + reload
sudo apachectl -t
sudo apachectl graceful

# 3. 옛 인프라로 복귀 확인
curl -sS https://bookmoa.com/storige-api/health
# uptime이 큰 값(옛 인프라)이면 복구 완료

컷오버 후 새 인프라에 들어간 데이터

롤백 시점까지 새 인프라에 들어간 세션/파일은 옛 인프라로 자동 동기화되지 않습니다. 이 경우 papascompany 인수팀이 다음 옵션 중 하나를 결정합니다:

  • 새 인프라의 새 세션 데이터를 옛 인프라로 수동 동기화
  • 해당 사용자에게 재편집 요청
  • 새 인프라가 받은 것 그대로 처리하고 옛 인프라는 fallback으로만 활용

자주 묻는 질문 (FAQ)

PHP 코드는 정말 한 줄도 안 바꿔도 되나요?
네. 모든 storige 호출이 STORIGE_API_URL 환경변수 또는 /storige-api/ proxy 경로를 거치도록 되어 있어, Apache vhost 두 줄만 새 주소로 바꾸면 PHP 측 호출 코드는 그대로 동작합니다. nimda의 storige_common.php도 getenv('STORIGE_API_URL')를 그대로 읽으므로 코드 변경 없이 새 URL을 사용합니다.
STORIGE_API_KEY도 바꿔야 하나요?
아니요. 기존 키 sk-storige-l3YVceH0sB739pgTfxRAxZAmLJROcMtgdKPIDYdVG0g를 새 인프라가 인식하도록 papascompany 측에서 미리 등록 완료했습니다. 변경 없이 그대로 사용하시면 됩니다. (보안상 키 갱신을 원하시면 papascompany 측에 요청 → 새 키 발급 → 양측 동시 교체)
webhook 수신 URL은 그대로 두어도 되나요?
네. webhook 수신처는 PHP 측이 결정하는 값이고, storige가 호출할 callbackUrl은 storige API 호출 시 PHP가 함께 보내는 파라미터입니다. 새 인프라도 같은 callbackUrl 파라미터를 받으므로 PHP 코드 변경 없이 동일하게 동작합니다.
webhook signature 검증 코드도 그대로?
네. X-Storige-Signature 헤더의 생성 알고리즘이 동일합니다 (옛/새 인프라 모두 base64 기반). 다만 향후 P6 단계에서 HMAC으로 강화할 계획이 있으며, 그때는 양측 협업으로 동시 변경 예정입니다.
다운타임이 발생하나요?
apachectl graceful은 진행 중인 요청은 옛 설정으로 마무리하고 새 요청부터 새 설정을 적용합니다. 따라서 다운타임 0초. 다만 변경 직후 잠시 동안 (아주 짧게) 옛 인프라로 가던 요청과 새 인프라로 가는 요청이 혼재할 수 있습니다. 이는 정상 동작이며, 두 인프라 모두 동일한 응답을 내므로 사용자 체감 영향 없습니다.
옛 storige(58.229.105.98:4000)는 언제 끄나요?
컷오버 후 24시간 무사고를 확인한 다음 papascompany 인수팀과 협의해서 결정합니다. 보통은 1주일 정도 fallback용으로 유지한 후 종료하는 방식이 안전합니다. 옛 storige를 끄기 전에 그 안의 데이터(파일, DB)를 새 인프라로 옮길지도 결정 필요.
새 인프라에서 합성된 PDF는 어디에 저장되나요?
새 인프라(Vultr Seoul VPS)의 Docker volume /app/storage/outputs/{jobId}/에 저장됩니다. nimda PHP가 받는 응답의 files.cover 등은 상대 경로(/storage/outputs/{jobId}/cover.pdf)이므로, 다운로드 시 새 storige base URL(https://api.papascompany.co.kr)을 prefix로 붙이면 됩니다. nimda의 기존 코드도 STORIGE_API_URL에서 base를 추출해 사용하므로 자동 처리됩니다.
테스트 페이지(bookmoa.noriter.co.kr/editor/worker-test.php)도 영향 받나요?
네, 같은 /storige-api/ proxy를 거치므로 자동으로 새 인프라와 통신하게 됩니다. worker-test.php에서 발행하는 잡은 새 인프라의 worker가 처리합니다.
CORS 문제는 없나요?
없습니다. PHP는 server-to-server 호출이라 CORS 무관. 브라우저 측 호출(에디터 JS 번들 → API)은 새 인프라가 *.bookmoa.noriter.co.kr 도메인을 자동 허용하도록 설정 가능합니다. 필요 시 papascompany 측에 도메인 알려주시면 CORS_ORIGIN에 추가합니다.
