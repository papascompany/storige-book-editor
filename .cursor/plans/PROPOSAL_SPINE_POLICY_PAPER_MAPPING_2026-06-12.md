## A. 신규 진입 책등 정책 — "제본 유형 × 템플릿 유형" 논리 충돌 해소

### 현행 사실
- 책등 공식: `spine = max(0, (pageCount/2) × 용지두께 + 제본여유)` (spine.service.ts:49)
- 제본 유형 DB 4종: `perfect`(무선 0.5mm) / `saddle`(중철 0.3mm, ≤64p, 4배수) / `spiral`(3.0mm) / `hardcover`(양장 2.0mm)
- 템플릿 유형: `flat-spread` = **책등 고정**(가드 구현됨) / `full`·`flat-spine` = **책등 가변**
- 파라미터 없는 신규 진입: 하드코딩 기본값(mojo_80g/perfect + 템플릿 기본 내지수) → 0.55mm 문제
- **검증 공백**: 상품(페이지 가변) ↔ flat-spread 템플릿(책등 고정) 연결을 막는 장치가 어디에도 없음

### 오너 제시 5유형 ↔ 현행 매핑(확인 요청)
| 오너 유형 | 제안 매핑 | 비고 |
|---|---|---|
| 소프트커버 | `perfect`(무선) | 현행 존재 |
| 하드커버 | `hardcover`(양장) | 현행 존재 |
| 중철 | `saddle` | 현행 존재. **책등≈0 정책 필요**(아래) |
| 압축앨범 | **신규 행** `album_compressed` | 합지 1장=1페이지라 **공식이 다름**: `pageCount × 합지두께 + 표지` — 현행 `(pageCount/2)×두께` 부적합 → binding_types 에 `perLeaf`(장당 계산) 플래그 1개 추가 필요. **인쇄소 합지 두께 스펙 필요** |
| 화보집 | **신규 행** `photobook` (hardcover 계열, margin 차등) | 인쇄소 스펙 필요 |

### 제안: 3계층 방어 (핵심 = "충돌은 등록 시점에 차단, 고객에겐 페이지수만 묻기")

**1계층 — 등록 시점 궁합 검증 (논리 오류의 근본 차단)**
- 템플릿셋의 책등 성격을 spread 템플릿 conversionMode 에서 **자동 파생**: `spineMode = fixed`(flat-spread) | `variable`(full/flat-spine). admin 목록·연결 화면에 뱃지 노출.
- **ProductTemplateSet 연결 생성/수정 시 검증 추가**(현재 공백):
  - `spineMode=fixed` 템플릿 ↔ **페이지수 고정 상품만** 허용(pageCountRange 단일값). 가변 상품 연결 시 거부.
  - `중철(saddle)` 상품 ↔ variable 템플릿 연결 시 경고("중철은 책등이 사실상 없음 — 펼침면 플랫형 권장").
- 효과: 운영자가 잘못 연결하는 순간 잡힘 — 고객 단계까지 오류가 내려가지 않음.

**2계층 — 진입 결정 트리 (오너의 "기본 룰" 구현 지점)**
```
/embed 진입
├─ ① 주문 파라미터 있음(bookmoa 정상 경로) ────────── 그대로 계산 (현행)
├─ ② 재편집(sessionId) ──────────────────────────── 세션 복원 우선순위 (구현됨)
├─ ③ 파라미터 없음 + spineMode=fixed ─────────────── 템플릿 고정 spine, 게이트 불필요 (가드 구현됨)
└─ ④ 파라미터 없음 + spineMode=variable ──────────── ★ 시작 게이트 모달
       제본 유형: 상품 연결(sortcode→ProductTemplateSet→상품)에서 역참조해 자동 결정
                   (역참조 불가 시에만 5유형 선택지 노출)
       페이지수: 고객 입력 (pageCountRange 클램프)
       → spine 계산 → 진입. 입력 거부 시 pageCountRange 최소값 폴백(0.55mm 임의값 제거)
```
- 오너 우려("선택 유형에 따라 논리 오류")는 ③에서 해소: **fixed 템플릿이면 제본/페이지수를 물을 필요 자체가 없음** — 책등은 디자인에 박제돼 있고, 페이지수는 인쇄 사양일 뿐 표지 지오메트리에 영향 없음. 게이트는 variable 일 때만 뜸.
- 중철 특례: `saddle` 선택 시 spine=0(또는 ≤1mm)으로 계산하고 표지에 책등 영역 비표시 — margin 음수 불가인 현행 공식 대신 binding_types 에 `spineVisible` 플래그 추가가 깔끔.

**3계층 — 런타임 가드 (대부분 구현 완료)**
- fixed: 재계산 차단(spineCalculator+SpreadPlugin 2중, e4eb328). variable: 페이지수 변경 시 자동 재계산(구현됨).
- 추가 1건: 세션 완료(주문 확정) 시 `세션 pageCount ↔ 상품 페이지수` 일치 soft 검증 → 불일치 경고를 주문 메타에 기록(검수 노출).

**구현 규모**: 1계층(API 검증+admin 뱃지) 소~중 / 2계층(게이트 모달+상품 역참조) 중 / binding_types 신규 2행+`perLeaf`·`spineVisible` 플래그 = 마이그레이션 1회. **선행 필요: 압축앨범·화보집 인쇄소 스펙(합지 두께, 양장 여유)**.

## B. 용지코드 매핑 — 기준과 방법

### 현행 사실
- 용지 마스터 = DB `paper_types`(code 조회, 시드 8종: `mojo_80g` 형식, 두께 mm/장 하드코딩)
- bookmoa 측 코드(`mojo100` 류)와 명명 불일치 → `SpineService.calculate` 가 **404 hard fail** → 편집기 폴백 없음(책등 미계산+콘솔 스팸 101회)
- admin 용지 관리 화면 없음(신규 용지 = DB 직접 조작), 두께 값의 인쇄소 스펙 대조 미실시

### 제안: "정본은 두께, 코드는 별칭으로 흡수"

**기준 1 — SSOT 와 명명 규칙 고정**: 정본은 Storige `paper_types` 테이블이며, 핵심 데이터는 **두께(mm/장)**. 정식 코드 규칙을 `<지종>_<평량>g`(소문자·언더스코어, 예: `mojo_100g`)로 문서화·동결.

**기준 2 — 호스트 코드는 alias 컬럼으로 흡수 (자동 추론 금지)**
- `paper_types.aliases` JSON 컬럼 추가(마이그레이션 1회): `mojo_100g.aliases = ["mojo100", "MOJO100", ...]`.
- 조회: 정규화(소문자·trim) → code 일치 → alias 일치 → **그래도 없으면 404 대신 200 + 폴백**:
  기본 용지(mojo_80g) 두께로 계산 + `warnings:[UNKNOWN_PAPER_TYPE]` 반환 → 편집기는 **경고 토스트 1회**(101회 콘솔 스팸 제거), 세션 metadata 에 미지 코드 기록(주문 검수 시 노출 — 인쇄 사고 방지선).
- `mojo100`→`mojo_100g` 같은 **유사도 자동 매칭은 채택하지 않음**: "모조 100g" 확신 불가 — 잘못 매칭된 두께는 책등 오차로 직결되므로 명시 alias 만 신뢰.

**기준 3 — 운영 절차**
1. **1회성**: bookmoa 실판매 용지 전수 목록(지종×평량×**두께** — 인쇄소 스펙서 기준)을 수집해 시드 확장 + alias 일괄 등록. ★ 두께 스펙서가 정본 — 현행 하드코딩 값(업계 추정치)도 이때 함께 검증.
2. **상시**: admin 에 용지 관리 화면(CRUD — GET API 는 이미 있음) 추가 → 신규 용지를 마이그레이션 없이 등록. 신규 지종 도입 시 "Storige 등록 먼저, bookmoa 노출 나중" 순서를 연동 규약에 명문화.
3. **조기 발견**: shop-session 발급 시점에 paperType 유효성 사전 검증 → 미지 코드면 bookmoa 에 명시 에러 반환(고객이 편집기에 진입하기 전에 연동 오류 발견).

**구현 규모**: API(aliases+폴백+정규화) 소 / admin 용지 CRUD 중 / 마이그레이션 1회. 선행: bookmoa 용지 목록+두께 스펙(오너/운영).
