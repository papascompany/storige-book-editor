2. Worker (PDF 검증·병합) API 엔드포인트

Method 경로 목적 필수
POST /api/worker-jobs/check-mergeable/external 저장 시점 dry-run — 병합 가능 여부 사전 검증 권장
POST /api/worker-jobs/synthesize/external 주문 시점 — PDF 병합 작업 큐잉 필수
GET /api/worker-jobs/external/{jobId} 작업 상태 폴링 (PENDING / PROCESSING / COMPLETED) 선택
POST {callbackUrl} (PHP 가 호스팅) Storige → 북모아 웹훅 — synthesis.completed/failed 필수

모든 외부 호출은 X-API-Key: {api-key} 헤더 필수. JWT 가 아닙니다.

4-3. Worker 병합 요청

❌ 원본 — 워커 호출 명시 없음
// 사용자가 업로드한 파일을 그대로 사용
// 별도 PDF 병합 단계 없음
$coverFile   = $_FILES['cover'];
$contentFile = $_FILES['content'];
moveAndStore($coverFile, $contentFile);
✅ 현재 — synthesize/external 호출
function requestSynthesis($orderSeqno, $sessionId,
    $coverFileId, $contentFileId, $spineWidth) {
  $ch = curl_init(STORIGE_API_URL .
    '/api/worker-jobs/synthesize/external');
  curl_setopt_array($ch, [
    CURLOPT_POST => true,
    CURLOPT_RETURNTRANSFER => true,
    CURLOPT_HTTPHEADER => [
      'X-API-Key: ' . STORIGE_API_KEY,
      // ⚠️ JWT 아님!
      'Content-Type: application/json',
    ],
    CURLOPT_POSTFIELDS => json_encode([
      'editSessionId' => $sessionId,
      'coverFileId'   => $coverFileId,
      'contentFileId' => $contentFileId,
      'spineWidth'    => $spineWidth,
      'orderId'       => 'ORD-' . $orderSeqno,
      'priority'      => 'high',
      'callbackUrl'   =>
        'https://bookmoa.com'
        . '/storige/proc/synthesis_callback.php',
    ]),
  ]);
  return json_decode(curl_exec($ch), true);
}

4-4. 웹훅 콜백 처리 (신규)

❌ 원본 — 콜백 없음
// 워커가 없으니 콜백도 없음
// PHP 가 직접 PDF 처리 가정
✅ 현재 — synthesis_callback.php 신규
// /storige/proc/synthesis_callback.php
<?php
$payload = json_decode(file_get_contents('php://input'), true);
$event   = $_SERVER['HTTP_X_STORIGE_EVENT'] ?? '';

if ($event === 'synthesis.completed') {
  $orderId       = $payload['orderId'];
  $outputFileUrl = $payload['outputFileUrl'];

  updateOrderFile($orderId, $outputFileUrl);
  updateOrderStatus($orderId, 210);  // 110 → 210
} elseif ($event === 'synthesis.failed') {
  notifyAdminOfSynthesisFailure($payload);
}

http_response_code(200);
echo json_encode(['received' => true]);

5. 인쇄물 사이즈 전달 방식 (3가지 옵션)

기존 가이드에는 pageCount/paperType/bindingType 만 있고 내지 사이즈 파라미터는 누락되어 있었습니다. 사이즈 전달 옵션 비교:

옵션 방식 PHP 측 작업 Storige 측 작업 적합 시나리오
A templateSetId 매핑 (상품, 사이즈) → templateSetId 매핑 테이블 관리 없음 사이즈 가짓수 적고 사전 등록 가능
B ?size=N (sizeno 인덱스) 폼 select → URL size 파라미터 이미 코드에 존재 — 가이드만 보충 상품별 사이즈가 정해져 있음 (북모아 product_sizes 테이블 사용)
C ?width=148&height=210 (mm) 폼 input(자유 입력) → URL width/height Product 엔티티/Admin/Editor 수정 (이번 PR 에서 완료) 자유 사이즈 입력 상품 (정사각형 크기 자유 등)

옵션 C 흐름 — 신규 구현

1
관리자: Storige Admin → 상품 편집 → "외부 쇼핑몰 사이즈 override 허용" Switch 활성화 → 저장
↓
2
PHP: 주문 폼에서 사용자가 사이즈 입력 → ?width=148&height=210 URL 생성
↓
3
Editor: Product API 응답의 allowCustomSize === true 확인 후 URL width/height 적용
↓
4
검증: 0 < w/h ≤ 2000mm, Number.isFinite — 미충족 시 무시 + console.warn (보안: 임의 사이즈 강제 방지)
↓
5
적용: useSettingsStore.setupProductBased 가 templateSet 의 사이즈를 customSize 로 덮어씀

옵션 C — PHP URL 생성 예시

✅ 옵션 C — width/height 직접 전달
// 사용자가 폼에서 가로 148mm, 세로 210mm 입력했다고 가정
$editorUrl = '/storige/edit.php'
    . '?productId='   . $productId
    . '&width='       . $userInputWidthMm   // 예: 148
    . '&height='      . $userInputHeightMm  // 예: 210
    . '&pageCount='   . $pageCount
    . '&paperType='   . $paperType
    . '&bindingType=' . $bindingType
    . '&return_url='  . urlencode('/mypage/order_detail.php');

// 주의: Storige Admin 에서 해당 상품의 allowCustomSize 가 true 여야 작동
//   - false 면 width/height 는 무시되고 templateSet 의 사이즈 사용

6. PHP 개발자 대응 우선순위

P0

P0
result.files 의 fileId 기반 처리로 콜백 코드 수정. URL 가정 시 NPE/오류.
P0
Worker API 인증을 API Key (X-API-Key) 로 변경. JWT 사용 시 401.
P0
/storige/proc/synthesis_callback.php 신규 구현. 병합 결과 수신 못 받으면 주문 상태 변경 불가.
P1
주문 처리 시 POST /synthesize/external 호출 추가. 안 하면 PDF 병합 안 됨.
P1
apiBaseUrl 명시 전달 (에디터 config). 빠지면 에디터가 잘못된 endpoint 시도.
P1
paperType, bindingType 코드 매핑 (북모아 폼 → URL 파라미터). 책등 자동 계산 정확도.
P2
onSave, onReady 콜백 활용 (자동 저장 진행률 UI). UX 개선.
P2
check-mergeable 호출로 저장 시점 사전 체크. 주문 직전 실패 방지.
P2
