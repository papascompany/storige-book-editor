# 북모아 Storige 통합 마이그레이션 현황 보고서 (2026-05-16)

본 보고서는 `storige` 프로젝트와 `bookmoa-mobile` 프로젝트의 통합 개발 이력을 분석하고, 초안부터 최종 결정 사항까지의 변경 과정을 정리한 문서입니다.

### 1.1 프로젝트 개요
기존 PHP 기반 인쇄 쇼핑몰에 현대적인 React 편집기, 관리자 콘솔, 그리고 자동화된 PDF 워커 서비스를 통합하여 서비스 전체를 플랫폼화하는 것을 목표로 합니다.

### 1.2 주요 타임라인 (파일 생성 및 작업 진행 순)
- **2025-12-03 ~ 12-07**: 프로젝트 초안(PRD v1.0~1.7) 작성. 기본 아키텍처(PHP+React+Node) 확립.
- **2026-03-24**: `bookmoa-mobile` 디자인 가이드 및 기본 모바일 UI 설계.
- **2026-04-08**: PHP 포팅 및 MariaDB 마이그레이션 가이드 작성 (현재 보류 상태).
- **2026-05-01 ~ 05-04**: 핵심 기능 구현 보고(Phase 2-9 완료). Sentry, E2E 테스트, 워커 합성 엔진 통합.
- **2026-05-06 ~ 05-10**: 멀티사이트 플랫폼화(Admin Platformization) 전략 수립 및 편집기 UX 개편.
- **2026-05-16 (오늘)**: 외부 사이트 연동 표준 계약(Phase 0) 최종 확정 및 회귀 테스트 체계 구축.

## 2. 단계별 변경 이력 분석

### 2.1 초안 (Draft)
- **주요 내용**: `docs/PRD.md` (2025-12)
- **핵심 목표**: 
  - 기존 PHP 쇼핑몰 유지 + 신규 편집기(React) + 관리자(React) + 워커(Node.js) 연동.
  - 워커를 클라이언트 설치형에서 서버 사이드 API 서비스로 전환 결정.
  - 편집기 반응형(모바일/데스크톱) 지원 필수화.
- **초기 설계**: 템플릿셋 기반의 데이터 구조 및 JWT 기반의 단순 인증 모델.

### 2.2 분석 (Analysis)
- **주요 내용**: `docs/INTEGRATION_AUDIT_2026-05-03.md`, `docs/01_SYSTEM_ARCHITECTURE_KR.md`
- **분석 결과**:
  - 기존 단일 사이트 중심 구조로는 여러 외부 사이트(bookmoa, mobile 등)를 수용하기 어려움 확인.
  - Webhook 서명 방식의 부재 및 CORS/CSP 정책의 유연성 부족 식별.
  - 편집기 소스(1,500줄)의 거대화로 인한 유지보수성 저하 분석.
- **대응 방향**: `sites` 테이블을 통한 동적 정책 관리 및 컴포넌트 분리 필요성 도출.

### 2.3 대응 내용 (Response & Implementation)
- **주요 내용**: `docs/MASTER_STATUS_2026-05-10.md`, `docs/PLATFORM_WORKER_INTEGRATION_v1.md`
- **구현 성과**:
  - **컴포넌트 분리**: `App.jsx`를 18개 이상의 독립 파일로 모듈화 완료.
  - **멀티사이트 지원**: `ApiKeyGuard` 및 `siteId` 주입 로직을 통해 여러 외부 사이트 식별 지원.
  - **편집기 모드 분리**: 고객용 '편집 모드'와 관리자용 '템플릿셋 수정 모드'를 시각적·로직적으로 격리.
  - **워커 엔진 고도화**: PDF 검증(Validate), 합성(Synthesize), 사전점검(Check-mergeable) API 안정화.

### 2.4 개선 사항 (Improvements)
- **주요 내용**: `docs2/storige_phase3_ui_체크리스트.md`, `docs/EDITOR_SCREENS.md`
- **UI/UX 개선**:
  - 객체 선택 핸들 개편(원형 코너, 캡슐 변), 하단 회전 핸들 도입으로 조작성 향상.
  - 사이드바 토글 시 캔버스 자동 중앙 정렬 로직 추가.
  - 템플릿셋별 '도구 메뉴 화이트리스트' 기능을 통해 상품별로 필요한 도구만 노출.
- **기술적 최적화**: 
  - snake_case와 camelCase를 모두 수용하는 URL 파라미터 호환 레이어 설계.
  - Supabase Storage를 활용한 이미지 업로드 헬퍼 및 썸네일 자동 생성 흐름 구축.

### 2.5 최종 결정 사항 (Final Decisions)
- **주요 내용**: `docs/PHASE_0_CONTRACT_DECISIONS_2026-05-16.md`
- **확정된 연동 표준**:
  1. **실행 모드**: "inline embed" 단일화 (iframe/오버레이 방식). 신규 탭이나 페이지 이동 지양.
  2. **Webhook 서명**: Base64 인코딩 (`identifier:event:timestamp`) 방식 유지. HMAC은 v2로 연기.
  3. **보안 정책**: `parentOrigin` 필수 지정. `targetOrigin='*'` 절대 금지.
  4. **데이터 케이스**: 외부 노출은 snake_case 권장, 내부는 camelCase 사용 (양방향 호환 지원).
  5. **다운로드**: 외부 클라이언트는 Storige API를 직접 호출하지 않고, 자체 서버 프록시를 통해서만 PDF 다운로드.
