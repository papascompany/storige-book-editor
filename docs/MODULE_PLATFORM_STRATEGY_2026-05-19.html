Storige 모듈 분기 + 플랫폼화 전략

기존 운영 영향 없이 SDK 레이어를 추가하여, 새 프로젝트가 SaaS / 자체 백엔드 / 완전 분리 3가지 모드 중 선택 가능한 하이브리드 구조 제안.

TL;DR — Phase A + C (SDK 분리 + Publish) 부터 시작. 운영 영향 0, 외부 즉시 사용 가능, Phase B (SaaS 강화) 로 자연 확장. 새 프로젝트 컨텍스트 받으면 모드 A/B/C 자동 권장.

§1 현재 자산 인벤토리 — 이미 70% 모듈화·플랫폼화 완료

Storige 는 이미 SaaS / 플랫폼 방향으로 진화 중. 본 전략은 그 기반 위에 SDK 레이어를 추가하는 것.

자산위치상태
Site / Tenant 모델apps/api/src/sites/✅ 구현됨 멀티사이트 지원 인프라
JWT siteId 자동 주입인증 미들웨어✅ 적용
CORS 동적 허용main.ts + sites.service✅ DB sites.allowed_origins 기반
외부 사이트 통합 v1docs/PLATFORM_INTEGRATION_v1.md✅ 문서화 어댑터 가이드
언어 중립 워커 가이드docs/PLATFORM_WORKER_INTEGRATION_v1.md✅ curl/Node/Python/Go 예시
PHP/bookmoa 통합docs/PHP_INTEGRATION_FINAL_v3.md✅ 운영 중
packages/canvas-corenpm 패키지 구조✅ 90% 모듈화 21개 플러그인
embed.tsx IIFE 번들vite.embed.config.ts✅ 빌드 가능
💡
결론: Storige 는 이미 "사내 SaaS + 플랫폼" 의 70% 인프라를 보유. 모듈 분기는 그 기반 위에 SDK 레이어를 더하는 것이 가장 자연스러움.

§2 5가지 분기 전략 비교

각 옵션의 영향·자유도·유지보수·부팅 시간 트레이드오프.

#전략영향 범위자유도동기·유지보수부팅 시간
1 완전 fork (새 리포지토리) 없음 ★★★★★ ❌ 양쪽 따로 fix 매우 김
2 monorepo 안 packages 확장 최소 ★★★★★ ⭐⭐⭐ 단일 소스 짧음
3 SaaS 멀티테넌트 중 ★★★★★ ⭐⭐⭐⭐⭐ 자동 매우 짧음
4 하이브리드 (SaaS 백엔드 + SDK 프론트) 최소~중 ★★★★★ ⭐⭐⭐⭐ 매우 짧음
5 Module Federation 높음 ★★★★★ ⭐⭐⭐⭐⭐ 짧음

§3 🌟 권장 — 하이브리드 (옵션 4 + 2 조합)

단일 옵션 대신 3-Layer 모듈화 가 가장 효율적. 다른 프로젝트가 자신의 상황에 맞춰 골라 쓸 수 있도록.

┌─────────────────────────────────────────────────────────────────────────┐ │ Storige Monorepo (현재 운영) │ │ │ │ ┌─ apps/api ────────────────────┐ ┌─ apps/admin ──────────────────┐ │ │ │ 멀티테넌트 SaaS 백엔드 │ │ 운영 admin (storige 전용) │ │ │ │ • sites 격리 (row-level) │ │ + 신규 site 관리 도구 │ │ │ │ • JWT siteId │ └────────────────────────────────┘ │ │ │ • CORS 동적 │ │ │ └─────────────────────────────────┘ ┌─ apps/editor ─────────────────┐ │ │ │ Storige.com 자체 편집기 │ │ │ └────────────────────────────────┘ │ │ │ │ ┌─ packages/ (SDK 레이어 — 외부 공개) ──────────────────────────┐ │ │ │ • canvas-core (npm public) │ │ │ │ • editor-react (npm public 또는 GitHub Packages) │ │ │ │ • editor-embed (IIFE bundle on CDN) │ │ │ │ • api-client (TypeScript SDK, 자동 생성) │ │ │ │ • types (공유 타입 + 이벤트 명세) │ │ │ └───────────────────────────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────────────────────────┘ ↓ 외부 통합 (3가지 모드) ┌─────────────────┼─────────────────┐ ↓ ↓ ↓ ┌──────────────────┐ ┌──────────────────┐ ┌──────────────────┐ │ 모드 A │ │ 모드 B │ │ 모드 C │ │ SaaS 풀버전 │ │ SDK+자체 API │ │ Fork+자체 호스팅 │ │ │ │ │ │ │ │ Storige API+UI │ │ Storige UI만 │ │ 완전 다른 도메인 │ │ 모두 사용 │ │ 가져가고 자체 │ │ (예: 명함, 메뉴) │ │ │ │ 백엔드 운영 │ │ │ │ 부팅: 1시간 │ │ 부팅: 1주 │ │ 부팅: 1개월 │ │ 자유도: 낮음 │ │ 자유도: 중간 │ │ 자유도: 최대 │ └──────────────────┘ └──────────────────┘ └──────────────────┘

왜 이 조합인가

🎯 현재 자산 100% 활용
sites 도메인 · CORS · JWT siteId 이미 있음. 그 위에 SDK 레이어만 추가.
🛡 기존 운영 영향 0
packages/ 폴더에 신규 추가만, apps/ 는 변경 없음.
🔓 자유도 단계적
모드 A→B→C 로 성숙 시 점진 이행 가능 (lock-in 없음).
⚡ 새 사이트 부팅 시간 최소
모드 A 면 site 등록만으로 즉시 출범.
🔁 모듈화 + 플랫폼화 동시
한 작업으로 두 가지 효과.

§4 단계별 로드맵

각 단계 후 영향 평가. 영향 0 인 Phase A · C 부터 시작 권장.

Phase A SDK 레이어 분리 ~10h ✅ 운영 영향 0 
storige/
  packages/
    canvas-core/         # 이미 90%, 정리만
+   editor-react/        # 신규 — apps/editor 에서 추출
+   editor-embed/        # 신규 — apps/editor/embed.tsx 이전
+   api-client/          # 신규 — TypeScript client (OpenAPI 기반)
    types/               # 기존 + 외부 공개용 정리

  apps/
    editor/              # 변경 없음 (점진 마이그레이션 가능)
    admin/               # 변경 없음
    api/                 # 변경 없음
완료 시점에 얻는 것:
  • 다른 프로젝트가 npm install 로 즉시 사용 가능 (모드 B)
  • IIFE 번들로 PHP/임의 스택 즉시 embed (모드 B 보조)
  • 기존 운영 100% 그대로
Phase B SaaS 강화 (모드 A 가능하게) ~15h ⚠ 중 영향 (테스트 필수) 
apps/api/
+ src/sites/
+   tenant-context.middleware.ts    # siteId 자동 주입
+   tenant-isolation.guard.ts       # 다른 site 리소스 접근 차단
+   site-onboarding.service.ts      # 새 site 자동 등록 + 토큰

+ src/middleware/
+   rate-limit-per-site.ts          # noisy neighbor 방지
+   usage-tracking.middleware.ts    # 사용량 미터링

apps/admin/
+ src/pages/Sites/Onboarding.tsx    # 신규 site 셋업 마법사
+ src/pages/Sites/Billing.tsx       # 사용량 + 결제 (옵션)
완료 시점에 얻는 것:
  • 새 사이트 = admin 에서 "사이트 추가" → 1시간 내 출범
  • Storige 가 진짜 SaaS 플랫폼
  • 운영 모니터링 site 별 분리
Phase C SDK Publish + 새 프로젝트 부팅 키트 ~8h ✅ 운영 영향 0 
release/
├── @storige/canvas-core    → npm publish (public)
├── @storige/editor-react   → npm publish
├── editor-embed.iife.js    → CDN (cdn.papascompany.co.kr/editor/v1/)
├── @storige/api-client     → npm publish
└── starters/
    ├── react-saas-template/        # 모드 A 부팅 키트
    ├── react-self-hosted/          # 모드 B 부팅 키트
    └── php-embed-template/         # 모드 B 부팅 키트 (PHP)
완료 시점: 누구든 5분 안에 새 프로젝트 부팅 가능
Phase D 새 프로젝트별 fork 가이드 (선택) ~3h ✅ 영향 0 (문서만)
완전 분리가 필요한 케이스(완전 다른 도메인)를 위한 fork 절차 + 동기 가이드 작성.
⏱
총 추정 시간: Phase A+C 만 = ~18h / 전체 (A+B+C+D) = ~36h

§5 의사결정 매트릭스 — 새 프로젝트 성격별 권장 모드

새 프로젝트가 무엇이냐에 따라 권장 모드가 달라집니다.

canvas-core 만 활용

§6 영향 평가 — "기존에 영향 없는가" 정밀 검증

Phase영향 항목영향 정도
A — SDK 분리apps/editor 그대로 + packages 신규 추가0
npm publish 시 storige 운영은 영향 없음0
B — SaaS 강화sites 도메인 강화 → 기본 site=storige 우선낮음
tenant isolation 미들웨어 잘못 적용 시 운영 영향 가능중
C — Publishnpm 패키지 자체 — storige 운영 영향 없음0
D — Fork 가이드문서만0
✅
결론: Phase A + C 는 영향 0. Phase B 만 신중한 검증 필요. 따라서 Phase A → C 먼저 (영향 0), Phase B 는 신규 site 등록 케이스 발생 시 별도 사이클로.

§7 SaaS 운영 모델 (Phase B 후)

라이선싱 + 과금 옵션

모델예시
무료 + 플랫폼 노출papascompany 브랜드 상품으로 자동 연결
월 정액site 별 $X/월
사용량 기반편집 세션 수, 합성 PDF 수, 스토리지 GB
B2B 계약 (현재 bookmoa 모델)사이트당 별도 계약
오픈소스 + 셀프호스팅 옵션Apache 2.0 → 누구나 fork

데이터 격리 옵션 (Phase B 세부)

레벨구현보안비용
Row-level (siteId 컬럼)현재 방향보통낮음
Schema-level (site 별 schema)미들좋음중간
DB instance-level (site 별 DB)멀리매우 좋음높음
💡
권장: Row-level 부터 시작 → 큰 site 만 분리 (점진 진화).
